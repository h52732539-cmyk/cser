# LiteVTR++ 多模型统一采样框架 — 方案概览

---

## 一、方案定位

在华为方核心 AI 模块完全不修改的前提下，通过纯调度层优化（采样策略、缓存复用、离线索引、智能路由），大幅降低多模型并行处理视频时的帧解码量、模型调用次数和功耗，同时保持甚至提升检索精度，多任务输出零退化。

## 二、技术路线（三层架构）

```
┌─────────────────────────────────────────────────────────┐
│  Layer 1: 离线索引层（充电/空闲时，一次性执行）           │
│    对图库中每个视频提取关键帧 → 调用视觉编码器一次        │
│    → 构建多粒度原型索引 → 缓存到本地                     │
│    同时预计算人脸时间线、场景标签、运动模式               │
├─────────────────────────────────────────────────────────┤
│  Layer 2: 查询路由层（每次用户查询，毫秒级）              │
│    仅调用一次文本编码器（~3ms）                           │
│    → 离线索引 numpy 检索（~0.7ms）                       │
│    → 智能路由判断查询难度：                               │
│       56% 简单查询 → 直接返回（零图像模型调用）           │
│       24% 中等查询 → 少量精定位                          │
│       20% 困难查询 → 完整两阶段推理                      │
├─────────────────────────────────────────────────────────┤
│  Layer 3: 多任务统一执行层（仅困难查询触发）              │
│    统一解码：5 个任务共享一条帧流，同帧只解码一次         │
│    跨任务缓存：同帧的模型输出跨任务复用                   │
│    元信息前置筛选：陀螺仪+帧差剔除无效帧                  │
│    所有模型视为黑盒 f(frame)→output，零修改               │
└─────────────────────────────────────────────────────────┘
```

## 三、使用的 5 个开源替代模型

| 替代对象 | 开源模型 | 来源 | 参数量 | 输入 | 输出 |
|---|---|---|---|---|---|
| 华为视觉编码器 | **MobileCLIP2-S0** | Apple, CVPR 2024 | 30M | RGB 图像 | 512 维特征向量 |
| 华为文本编码器 | **MobileCLIP2-S0** | 同上 | 同上 | 文本字符串 | 512 维特征向量 |
| 华为高光检测 | **MomentDETR** | Lei et al., NeurIPS 2021 | ~150M (含 CLIP ViT-B/32) | 视频帧序列 | 每帧显著性分数 + 时间段预测 |
| 华为人脸检测 | **InsightFace SCRFD-10GF** | DeepInsight, buffalo_l 套件 | ~34M | RGB 图像 | 是否有人脸 + 置信度 |
| 华为人脸识别 | **InsightFace ArcFace-r50** | 同上 | ~65M | RGB 图像 | 512 维人脸特征向量 |
| 华为场景分类 | **MobileNetV3-Small** | Google, ImageNet-1k 预训练 | 2.5M | RGB 图像 | 场景类别标签 |

> 以上模型在框架中均视为黑盒函数，仅通过标准接口调用（encode / detect / embed / classify），不做任何权重修改、量化或微调。华为方替换时只需实现相同接口。

## 四、核心性能数据

### 测试环境

- 数据集：MSR-VTT（YouTube 真实视频），1000 视频库 × 1000 查询
- 硬件：RTX GPU + CUDA（模拟端侧 NPU 推理）
- 5 个开源模型同时运行

### 效率提升

| 维度 | 传统方案（各模型独立） | LiteVTR++ | 节省 |
|---|---|---|---|
| 每视频帧解码量 | 189 帧 | **47 帧** | **-75%** |
| 每视频模型调用次数 | 1577 次 | **1187 次** | **-25%** |
| 每查询检索延迟 | ~800 ms | **0.74 ms** | **1080 倍加速** |
| NPU 等效功耗 | ~2.4W 持续 | **< 0.5W** | **-80%** |

### 精度保持

| 维度 | 传统方案 | LiteVTR++ | 变化 |
|---|---|---|---|
| 视频检索 R@1 | 33.4% | **38.8%** | **+5.4pp（不降反升）** |
| 视频检索 R@5 | ~56% | **61.0%** | +5pp |
| 多任务模型输出 | 基准 | **100% 字节级一致** | 零退化 |

### 安全性（元信息噪声场景）

| 元信息质量 | 传统规则方案 R@1 | LiteVTR++ R@1 | 传统 GT 误删率 | LiteVTR++ GT 误删率 |
|---|---|---|---|---|
| 完美元信息 | 57.7% | 51.3% | 10.7% | **0%** |
| 中等噪声 | 41.7% | **46.7%** | 14.3% | **0%** |
| 冲突元信息 | 25.7% | **32.3%** | **49.3%** | **0%** |

> 传统规则方案在元信息有噪声时精度急剧下降甚至低于纯语义基线，且误删率高达 49%。LiteVTR++ 的 calibrated safety 机制自动拒绝不可靠的过滤操作，在所有噪声场景下 GT 误删率均为 0%。

### 视频库规模扩展性

| 视频库规模 | 索引大小 | 查询延迟（已测/估算） |
|---|---|---|
| 1,000 视频 | ~12 MB | **0.74 ms**（已测） |
| 10,000 视频 | ~120 MB | ~7.4 ms（线性估算） |
| 100,000 视频 | ~1.2 GB | ~50 ms（需接 HNSW 近似索引） |

## 五、华为方对接方式

华为方替换开源模型时，只需实现以下标准接口：

```
encode_frames(images)  → 视觉特征向量
encode_text(texts)     → 文本特征向量
detect(images)         → 人脸检测结果
embed(images)          → 人脸识别特征
score(images)          → 高光显著性分数
classify(images)       → 场景分类标签
```

框架代码零修改，新建模型适配文件即可完成替换。
